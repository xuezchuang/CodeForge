# Git Commit Message

## Workflow

1. Inspect the worktree before writing:
   - Use `git/status` with default arguments.
   - Use `git/diff` with `stat=true` for unstaged changes.
   - Use `git/diff` with `cached=true` and `stat=true` for staged changes.
   - Read the relevant `git/diff` output, using `cached=true` for staged changes; for untracked files, inspect file names and contents when they affect the message.

2. Separate intent:
   - Identify whether changes form one logical commit or multiple commits.
   - Do not stage, unstage, revert, or discard files unless the user asked.
   - If unrelated changes are present, either draft separate messages or ask which scope to commit.

3. Draft the message:
   - Prefer `type(scope): summary` when a clear Conventional Commit type exists.
   - Keep the summary imperative and under 72 characters.
   - Add a body when the diff has meaningful rationale, risk, tests, or multiple bullets.
   - Mention tests or verification only when evidence exists.
   - Avoid "generated by AI" boilerplate.

4. Commit only when requested:
   - Confirm the staged file set matches the intended commit.
   - If nothing is staged and the user clearly asked to commit all relevant dirty work, stage the intended files explicitly with `git/add`.
   - Use `git/commit` with `message`, `body`, or `body_paragraphs`.
   - Report the resulting commit hash and exact message.

## Message Patterns

Use a short message for a cohesive small change:

```text
fix(composer): align slash command menu
```

Use a body for larger work:

```text
feat(mcp): add shared MCP runtime controls

- Keep MCP server state in app state instead of per-run startup
- Add connect and disconnect commands for configured servers
- Refresh model tools after MCP management tool calls

Verified with npm run build and cargo test mcp --lib.
```

For mixed changes, propose a split:

```text
Suggested split:
1. feat(mcp): add shared MCP runtime controls
2. fix(composer): align slash command menu
```

## Safety Rules

- Preserve unrelated dirty work.
- Do not claim tests ran unless command output proves it.
- Do not include secrets, tokens, or private file contents in the message.
- If the diff includes generated build artifacts, call that out before committing.
